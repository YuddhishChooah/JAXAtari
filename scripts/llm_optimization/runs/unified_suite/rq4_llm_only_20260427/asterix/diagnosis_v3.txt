# Diagnosis: Asterix Parametric Policy

## 1. Root cause
The policy under-collects (≈10 pickups/1000 frames, only 0.5 collectibles visible on average) AND dies to all 3 lives via diagonal lane transitions. The core defect is that lane-entry safety is judged only at *arrival time* using a single `k_steps` estimate and only against the lane's enemy at `target_x`, while the player physically passes through intermediate y-positions of the destination lane during diagonal motion. Combined with a mismatched danger scale (hard gate uses `danger_threshold=22` as an x-gap but `danger_radius=32` for soft danger), the agent confidently steps diagonally into lanes whose enemy is closing in.

## 2. Evidence from metrics
- `hits_taken=3`, `terminal_game_over=1`, `lives_remaining=0` — survival is the binding constraint.
- `hit_diagonal_fraction=1.0`, `hit_horizontal_fraction=0.0` — every death is on a diagonal lane change, confirming unsafe transitions.
- `avg_hit_enemy_gap=18.4` — collisions occur at gaps below `danger_threshold=22`, meaning the gate is being passed yet collision still happens (projection horizon too short or wrong x).
- `avg_collectibles_visible=0.5` — agent rarely sees items; it likely sits in a lane that is "safe but empty" rather than scouting.
- `score_rate_per_1000=499` and `item_pickup_rate_per_1000≈10` — sparse harvesting.
- `horizontal_rate=0.83`, `vertical_rate=0.009` — the agent commits to lateral patrol far more than lane-changing; when it does change lanes (16% diagonal), those moves are the lethal ones.
- `avg_stage_index=3.2`, `topmost=0`, `bottommost=7` — full lane range is reachable; vertical mobility is not the problem.

## 3. Policy logic causing it
- **Single-point arrival check**: `_lane_min_enemy_gap` evaluates only at `k_steps` and only at `target_x`. During a diagonal move, the player crosses the destination lane while the enemy is still moving; transit-window danger isn't modeled.
- **Inconsistent danger scales**: `danger_radius=32` defines soft danger normalization, but the hard gate uses `danger_threshold=22` directly as a pixel gap. The two are not coupled, so tuning one silently breaks the other.
- **Crude `k_steps = 2 + 4*d`** ignores horizontal travel time inside a lane; the projection used for the gate underestimates enemy displacement when the player has to traverse lateral distance after arriving.
- **Patrol fallback dominates when no eligible lane**: when collectibles aren't visible (very common, given 0.5 avg), the fallback lane scoring uses only danger + small lane_dist — it pulls the agent toward the safest *current* lane and then patrols laterally, which is exactly the "avg_collectibles_visible≈0.5" symptom (not actively repositioning to where new items spawn).
- **Escape logic uses `safe_lane[adjacent]`** which is the same arrival-gate booleans — if those gates are wrong, escapes are also unsafe.
- **No recent-pickup memory / no persistent commitment**: every frame re-picks a lane; combined with lateral-only motion 83% of the time, the agent oscillates and rarely commits to crossing.
- **Value handling**: `value_weight=1.2` on a 50–300 scale dwarfs `lane_dist_penalty=12`/lane and `x_intercept_penalty=0.25`/pixel. A 300-pt item far away always wins, even when an enemy will arrive first — making long diagonal commits routine. This explains why diagonals are lethal.

## 4. What to preserve
- The compact 6-parameter shape and the lane-aligned indexing.
- Two-frame dx estimation for both enemies and collectibles.
- Value table by `visual_id` (correct concept).
- Reachability cost via `lane_dist`.
- The two-tier structure: targeted collectible mode + safety fallback + immediate-danger escape.
- 100% win rate / 1703 score baseline shows the skeleton works; only the safety model and idle behavior need surgery.

## 5. What to change next
- **Transit-window safety, not arrival-point safety**: for any candidate lane, evaluate enemy gap over a *range* of k (e.g., min over k=1..k_arrival) at the player's evolving x, not just `target_x` at `k_arrival`. Reject lanes whose minimum gap during transit falls below a buffer.
- **Couple the two danger scales**: derive the hard gate from `danger_radius` (e.g., gate = `danger_radius - margin`) so a single knob controls safety. Also widen the buffer for *destination* lanes during diagonals vs. staying put.
- **Per-step motion model**: replace `2 + 4*d` with one consistent rate (player_dx_per_frame ~1, lane change ~ a few frames each), and use it both to project enemy x and to weight value by time-to-pickup.
- **Time-discount value, don't just penalize distance linearly**: score ≈ `value / (1 + α·arrival_steps)` minus danger. This naturally suppresses far-lane 300-pt suicide commits without needing a huge `lane_dist_penalty`.
- **Pre-emptive vertical movement when no items are visible**: the fallback should *scout* by drifting toward lanes likely to spawn items (e.g., lanes with no enemy AND not the current lane), not just minimize danger in place. Avg 0.5 visible collectibles means the agent is probably in a frame where one *will* appear soon — being at the edge lanes when they spawn is what raises pickup rate.
- **Abort-in-flight**: while executing a diagonal, re-check the destination lane's transit gap each frame and fall back to a pure horizontal away-from-enemy action if the gap collapses. Currently the policy commits and the only re-evaluation is the immediate-danger escape on the *current* lane.
- **Symmetric escape**: when in danger, also check that the diagonal escape lane's enemy isn't itself closing at small projected gap; prefer pure LEFT/RIGHT away from enemy if both adjacent lanes fail transit safety.
- **Reduce idle lateral patrol toward screen center**: replace with movement toward the nearest lane that has historically (last frame) shown a collectible or toward the lane edge opposite the nearest enemy. The 83% horizontal rate suggests the agent is wasting frames laterally instead of repositioning vertically to harvest.
- **Parameter additions to consider** (still compact): `transit_buffer`, `arrival_time_discount`, `escape_buffer` — but fold redundant ones (e.g., merge `danger_threshold` into `danger_radius`-derived gate).
